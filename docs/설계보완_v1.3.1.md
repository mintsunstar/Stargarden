### 2.1 `006_align_phase2_tables.sql` (최종)

> 파일: `supabase/migrations/006_align_phase2_tables.sql`  
> v1.3.1 긴급 수정(2026-07-09): profiles nullable 보정, 정책 문법 수정 반영.

```sql
-- §0 profiles 안전 보정 (구버전 스키마 대비)
alter table profiles alter column birth_date drop not null;
alter table profiles add column if not exists onboarding_completed_at timestamptz;
alter table profiles add column if not exists total_blooms int not null default 0;

-- §1 notification_settings 컬럼 3종
-- §2 push_tokens device_id + drop/create policy (IF NOT EXISTS 사용 금지)
-- §3 premium_monthly index + grant_source event
-- §4 handle_new_user() → profiles + notification_settings
-- §5 notification_settings backfill
```

전체 SQL은 마이그레이션 파일을 참조. PostgreSQL은 `CREATE POLICY IF NOT EXISTS`를 지원하지 않으므로 `DROP POLICY IF EXISTS` + `CREATE POLICY` 패턴을 사용합니다.

**001 현재 레포 기준 참고** (2026-07-09):
- `birth_date` — 이미 nullable (`date` only, NOT NULL 없음)
- `onboarding_completed_at`, `total_blooms` — 001에 이미 존재  
→ §0 보정은 idempotent 안전망이며, 신규 클론에는 필수 버그 수정이 아닐 수 있음. **006의 실질 변경은 notification_settings 트리거·push_tokens 정책·grant_source event**.

## 3. 웰컴 vs 일일 씨앗 — Partial Unique Index (공식 스펙)

v1.1의 `unique(user_id, granted_date, grant_source)` composite 방식은 **폐기**합니다.

```sql
-- daily: KST 기준 하루 1회 (FR-C01)
create unique index idx_user_seeds_daily_unique
  on user_seeds (user_id, granted_date)
  where grant_source = 'daily';

-- welcome: 평생 1회 (UC-01)
create unique index idx_user_seeds_welcome_unique
  on user_seeds (user_id)
  where grant_source = 'welcome';
```

- **FR-C01 문구 수정**: "1일 1개, KST 00:00 리셋" → **`grant_source='daily'` 기준 1일 1개**
- **UC-01 각주**: 웰컴 씨앗은 당일 일일 씨앗과 별개로 지급·인벤토리 누적
- **`event`**: 시즌 확정 시 `where grant_source = 'event' and granted_date = ?` partial index 추가

### user_seeds INSERT 정책 (의도적 설계)

> user_seeds INSERT는 클라이언트 RLS 정책 없음 = **service_role 전용**.  
> 생성 경로: `grant-daily-seed`, `complete-onboarding`(welcome), 구독 혜택(`premium_monthly`), 향후 이벤트(`event`).

### collections INSERT 정책

> collections INSERT는 **`confirm-bloom` service_role 전용**. 클라이언트가 도감을 직접 조작할 수 없음.

## 4. RLS 강화 — `007_rpc_rls_hardening.sql`

**원칙**: 클라이언트는 정원·프로필 민감 필드를 **읽기 전용**. 쓰기는 Edge Function(service_role) 경유.

### 4.1 plants / user_seeds

```sql
drop policy if exists "plants_insert_own" on plants;
drop policy if exists "plants_update_own" on plants;
drop policy if exists "user_seeds_update_own" on user_seeds;
-- plants_select_own, user_seeds_select_own 유지
```

| 액션 | 허용 주체 |
|---|---|
| plants SELECT | 클라이언트 (본인) |
| plants INSERT/UPDATE | `plant-seed`, `water-plant`, `tap-starlight`, `confirm-bloom`, `wake-garden` |
| user_seeds UPDATE (inventory→planted) | `plant-seed`, `grant-daily-seed`, `complete-onboarding` |

### 4.2 profiles — 민감 필드 UPDATE 차단

현재 `profiles_update_own`은 `last_active_at`, `total_blooms`, `garden_slots`, `is_sleeping`, `ohang_type` 등을 클라이언트가 직접 조작할 수 있는 구멍이 있습니다.

```sql
drop policy if exists "profiles_update_own" on profiles;
-- profiles_select_own 유지
```

| 필드 | 갱신 주체 |
|---|---|
| `display_name`, `birth_*` (월 1회) | `update-profile` Edge Function |
| `ohang_type` (재산출) | `complete-onboarding`, `update-profile` |
| `onboarding_completed_at` | `complete-onboarding` |
| `last_active_at` | `touch-active` |
| `is_sleeping` | `touch-active`, `wake-garden` |
| `total_blooms`, `garden_slots` | `confirm-bloom` |
| `profile_edit_count_this_month`, `profile_edit_month` | `update-profile` |

## 6. Edge Function 상세 스펙

### 6.1 성장 수치 공통 (`_shared/gameConstants.ts`)

클라이언트 `constants/game.ts`와 동일 값을 서버에서 참조합니다.

| 상수 | 값 |
|---|---|
| `CARE_BONUS.watering` | 6 |
| `CARE_BONUS.starlight` | 0.5 (10탭당) |
| `CARE_BONUS.starlightDailyCap` | 2 |
| `GROWTH_HOURS.normal` | 48 |
| `GROWTH_HOURS.rare` | 96 |
| `SLEEP_THRESHOLD_DAYS` | 30 |
| `GARDEN_START_SLOTS` | 6 |
| `GARDEN_MAX_SLOTS` | 12 |
| `RARE_PROBABILITY` | 0.15 |

### 6.2 plants ↔ seeds JOIN (FR-C04)

`plants`에는 `required_hours`가 없습니다 (정규화). 아래 4개 함수 모두 동일 JOIN을 사용합니다.

```
plants.user_seed_id → user_seeds.seed_id → seeds.required_hours, rarity, ...
```

클라이언트가 보유한 `requiredHours` 등은 **신뢰하지 않습니다**.

### 6.3 `starlight_date` 형식

KST 기준 `YYYY-MM-DD` (`Asia/Seoul`). 서버에서 변환·포맷하며 클라이언트 로컬 타임존에 의존하지 않습니다.

### 6.4 `confirm-bloom` 부수 효과

1. 서버에서 성장 진행률 재계산 → 100% 확인
2. `plants.growth_stage = 2`, `plants.bloomed_at = now()`
3. `collections` INSERT (`unique(user_id, seed_id)`)
4. `profiles.total_blooms += 1`
5. `total_blooms % 5 == 0`이면 `profiles.garden_slots += 1` (최대 12)

> `profiles.garden_slots`, `profiles.total_blooms`는 `001`에 이미 존재. 클라이언트 Zustand와 이중 관리되므로 서버가 단일 진실 공급원(SSOT)이 됩니다.

### 6.5 `complete-onboarding`

| 항목 | 내용 |
|---|---|
| 호출 시점 | SCR-003 오행 결과 확인 후, **첫 정원 진입 전** 1회 |
| 중복 방지 | `onboarding_completed_at IS NOT NULL`이면 거부 |
| profiles 갱신 | birth_*, ohang_type, `onboarding_completed_at = now()` |
| 웰컴 씨앗 | `grant_source='welcome'`, `idx_user_seeds_welcome_unique`로 평생 1회 |
| 웰컴 종 선택 | 사용자 `ohang_type`에 매칭되는 **일반(normal) 씨앗 1종** 랜덤 (에너지 코드 일치) |
| 응답 | `{ ohang_type, user_seed_id, seed_id, species_name }` |
| 클라이언트 | 튜토리얼(SCR-004)은 서버 지급 씨앗으로 연출만 수행 |

개발자 모드(`VITE_DEV_BYPASS_AUTH`)에서는 호출 생략 가능. `VITE_USE_SERVER_GARDEN=true`일 때만 서버 경유.

### 6.6 `touch-active` / `wake-garden` (FR-C07)

**잠듦/깨어남은 2단계 API로 분리**합니다. (모달 확인 UX 유지)

#### touch-active (세션 시작 1회)

1. **갱신 전** `profiles.last_active_at` 읽기
2. 이전 접속과의 갭 ≥ 30일이면:
   - `profiles.is_sleeping = true`
   - 해당 유저 `plants.is_sleeping = true` (일괄)
   - 응답 `{ needsWakeModal: true }`
3. `profiles.last_active_at = now()` 갱신
4. 그 외 `{ needsWakeModal: false }`

#### wake-garden (WakeUpModal 확인 시)

1. `profiles.is_sleeping = false`
2. `plants.is_sleeping = false` (일괄)
3. 응답 `{ awakened: true }`

### 6.7 `update-profile` (SCR-304)

| 항목 | 내용 |
|---|---|
| 입력 | display_name, birth_date, birth_time_unknown, birth_time, is_lunar |
| 검증 | `profile_edit_count_this_month < 1` (월 단위 KST) |
| 처리 | birth 변경 시 ohang_type 재산출, edit_count 증가 |
| RLS | profiles 직접 UPDATE 불가 → 본 함수만 |

### 6.8 `grant-daily-seed`

- KST `granted_date` 기준 `grant_source='daily'` partial unique 검증
- 15% 희귀 확률 추첨을 **서버에서 수행** (클라이언트 `pickDailySeed()` 제거 또는 미리보기 전용)
- 오늘의 `daily_energies.energy_code`와 연동

## 8. 씨앗 ID 통일 (Phase 2-A 의사결정)

| | A안 (text PK) | B안 (UUID PK) **권장** |
|---|---|---|
| 내용 | DB seeds.id를 `b001` 형태로 통일 | 클라이언트가 서버 UUID 참조 |
| 장점 | 목업 수정 최소 | sounds/plants/collections FK와 일관 |
| 단점 | 전 테이블 중 유일한 예외 PK | `constants/seeds.ts` ID 전면 교체 |

**현황**: DB `003_seed_data.sql`은 `b0000001-0000-0000-0000-000000001` 등 **고정 UUID**. 클라이언트는 `b001~b00f` 단축 ID 사용 중.

**조치 (B안 확정 시)**: `constants/seeds.ts`의 id를 003 UUID와 1:1 매핑. Edge Function 연동 전 필수.

## 9. 크리티컬 패스 (Phase 2)

```
2-A (스키마 006/007 + 씨앗 ID 통일)
  → 2-C (안티치트 EF 3종 + touch-active/wake-garden/complete-onboarding)
    → 2-C′ (gardenStore → invokeEdgeFunction 전환, VITE_USE_SERVER_GARDEN)
      → 2-B (Auth OAuth + update-profile)
        → 2-D (grant-daily-seed, confirm-bloom, …)
          → 2-E (SCR-303 알림)
            → 2-F (결제·legal_consents·delete-account)
```

| Phase | 기간 | 완료 기준 |
|---|---|---|
| 2-A | 2~3일 | `supabase db diff` 없음, SCR-303 필드 1:1 |
| 2-C | 5~7일 | JWT로 plants 직접 UPDATE 시 거부 |
| 2-C′ | 2~3일 | `VITE_USE_SERVER_GARDEN=true`에서 정원 루프 동작 |
| 2-B | 3~5일 | 가입→004→complete-onboarding→DB 반영 |
| 2-D | 5~7일 | 일일 씨앗·개화 서버 검증 |
| 2-E | 4~5일 | push_tokens upsert + SCR-303 |
| 2-F | 별도 | verify-purchase, legal_consents, Storage 정리 |

목업(Zustand only)은 **2-C´ 완료 전**까지 `VITE_USE_SERVER_GARDEN=false`(기본)로 유지.

### `.env.example` 추가

```bash
VITE_USE_SERVER_GARDEN=false   # true: gardenStore가 Edge Function 호출
```
